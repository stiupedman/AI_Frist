# 家教平台 ABC 优化设计

## 目标

在现有最小业务闭环上，按 A（上线稳定性）、B（使用体验）、C（代码结构）的顺序完成一轮成品化优化。保留模拟支付，不接入第三方支付、短信、微信或小程序，不增加依赖，不改变现有 API 路径和数据库字段含义。

完成标准：家长、教员、管理员的关键业务链可运行；越权、非法金额、重复提交和非法状态迁移被后端拦截；桌面和手机视口可完成主要操作；前后端构建与现有自动检查通过。

## 实施路线

采用纵向加固：每个阶段先检查现状，再做最小修改并验证。A 阶段先建立可信的业务边界，B 阶段改善这些业务的操作体验，C 阶段只拆分已经形成独立职责的代码。三个阶段分别提交，便于定位回归。

## A：上线稳定性

### 权限与数据归属

- 控制器继续使用现有 `@RequiresPermissions`，检查所有接口是否有与角色配置一致的权限标识。
- 服务层负责数据级权限：家长只能操作自己发布需求形成的订单，教员只能操作自己承接的订单，管理员操作必须经过管理权限入口。
- 黑名单、作业、投诉、通知、课时、付款等按资源 ID 操作的接口，在写入前校验当前用户与资源的关系。
- 前端按钮显隐仅用于体验，不能替代后端权限检查。

### 状态与幂等

- 需求、订单、邀请、作业、付款、退款、课时确认和结算沿用现有状态值。
- 状态变更使用带当前状态条件的更新语句；更新行数不是 1 时返回“状态已变化，请刷新后重试”。
- 重复点击、重复请求和过期页面只能让第一次合法操作成功。
- 不引入状态机框架，状态规则继续集中在现有状态常量和服务校验中。

### 财务一致性

- 支付、模拟支付、付款审核、退款、对账、开票、课时确认和结算等多表写入使用事务。
- 金额必须大于 0、最多两位小数，并受数据库字段精度上限约束；退款金额不得超过原付款金额。
- 平台抽成和实收金额只由后端计算，不接受前端计算结果。
- 付款记录、财务流水和通知在同一事务中提交；失败时全部回滚。

### 安全检查

- 使用 `security-scan` 做一次标准扫描，重点检查鉴权、对象级越权、SQL 注入、敏感数据返回和财务一致性。
- 只修复有代码证据且可复现的高风险问题；其余发现记录为后续事项，不扩大本轮范围。

### A 阶段验证

- 角色权限与 SQL 权限脚本检查。
- 合法和非法状态迁移、重复操作、对象归属、金额精度和退款边界检查。
- 后端模块编译及已有测试。

## B：使用体验

### 响应式工作台

- 桌面端保留现有工作台、指标、模块导航和表格布局。
- 手机端将长模块导航替换为现有 Element UI 选择器，减少首屏占用。
- 表格继续使用横向滚动，不维护第二套移动端页面；状态、金额和操作列优先可见。
- 弹窗宽度在窄屏限制为约 `94vw`，表单项改为单列，底部按钮不得溢出或遮挡。

### 操作反馈

- 列表提供加载、空数据和失败反馈。
- 关键提交按钮在请求期间禁用并显示处理中状态，避免重复提交。
- 成功后关闭弹窗并刷新当前模块；失败时保留用户输入，由现有全局请求错误处理展示原因。

### 重点流程

- 家长：发布需求、查看订单、模拟支付、提交作业、评价或投诉。
- 教员：申请或接受预约、登记课时、布置作业、反馈作业、完成订单。
- 管理员：审核付款、退款与对账、管理黑名单、处理投诉。

### B 阶段验证

- 前端单元自检与生产构建。
- 使用 Playwright 检查桌面和手机视口的页面身份、非空渲染、控制台、关键交互和截图。
- 未启动完整后端数据环境时，明确记录无法覆盖的真实数据流程，不用构建成功替代页面验证。

## C：代码结构

### 审计原则

- 先用 `ponytail-audit` 和架构技能生成复杂度清单，再按收益和风险选择修改。
- 优先删除重复、提取纯逻辑和移动已形成边界的代码；不新增单实现接口、工厂或转发层。

### 后端边界

- 将管理员专属路由移动到 `TutoringAdminController`，保持 `/tutoring/**` 路径、权限和返回格式不变。
- 将支付、退款、对账、发票、财务流水和结算提取到独立财务服务，使事务边界集中。
- 公共资源归属校验只保留一份；若提取财务服务会造成校验或通知逻辑复制，则保留在原服务中，以较少重复为准。
- Mapper 暂不强拆；只有在不复制 SQL、不改变映射的情况下能明显降低耦合时才拆分。

### 前端边界

- 提取管理员专属工作台及其状态、查询和操作，主页面保留角色切换与家长/教员公共订单流程。
- 纯状态标签和格式化逻辑可移到已有配置或独立纯函数文件，并保留小型可运行检查。
- 不把每个弹窗拆成单独组件；只有拥有独立数据加载和提交生命周期的区域才提取。

### C 阶段验证

- 拆分前后执行同一组后端测试、前端检查和构建。
- 使用 `git diff --check` 检查格式，并核对 API 路径、权限标识、SQL 映射和角色配置无意外变化。

## 错误处理

- 参数或业务状态错误继续抛出项目现有 `ServiceException`，返回可直接理解的中文提示。
- 数据库条件更新失败视为并发或重复操作，不静默成功。
- 财务事务失败不得留下孤立付款、流水、结算或通知记录。
- 前端不吞掉业务错误；用户取消确认框不作为错误提示。

## 不做

- 不接真实支付、退款网关、短信、微信、邮件或发票平台。
- 不开发原生 App 或小程序。
- 不引入新的 UI、状态管理、状态机或测试框架。
- 不重写 RuoYi 基础用户、角色、网关和权限体系。
- 不对与家教业务闭环无关的模块做仓库级重构。

## 最终验收

1. A、B、C 按顺序完成并分别验证。
2. 三种角色的主要操作权限与对象归属正确。
3. 重复操作和非法状态迁移不会产生重复业务记录。
4. 金额、抽成、退款、流水和结算保持一致。
5. 桌面与手机视口没有阻断主流程的溢出、遮挡或不可点击控件。
6. 后端编译、前端检查、生产构建和 `git diff --check` 通过。
